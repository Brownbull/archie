# Story 10-2: Blueprint Library Expansion

## Status: review
## Epic: Epic 10 -- Canvas & Content Enhancements

## Intent
**Epic Handle:** "More to see, more to start from"
**Story Handle:** "This story gives users 8 more starting points — more architectures to explore means more trade-offs to learn"

## Overview

As a user,
I want 8 additional blueprint architectures covering different domains and patterns,
So that I have more starting points for exploration and more trade-off patterns to learn from.

**FRs:** FR88
**Size:** MEDIUM (4 pts) -- 3 tasks, ~12 subtasks, 8 files
**Risk Flags:** DATA_PIPELINE
**Depends on:** None (uses existing blueprint schema and component library)

## Acceptance Criteria

**AC-1: 8 New Blueprints Created (FR88)**
Given the existing 2 blueprints (WhatsApp, Telegram),
When the 8 new blueprints are added,
Then the blueprint library contains 10 total, each with unique architecture patterns.

**AC-2: Schema Compliance**
Given each new blueprint YAML,
When validated against BlueprintSchema,
Then all pass without errors. Every component reference resolves to an existing library component.

**AC-3: Diverse Architecture Patterns**
Given the 8 new blueprints,
When browsed as a collection,
Then they cover at least 5 distinct architectural patterns (request-response, pub-sub, caching-heavy, reverse-proxy, push-based, queue-based).

**AC-4: Tier Annotations**
Given each new blueprint,
When tier is evaluated,
Then each has a valid tier annotation reflecting its completeness (number of components, category spread).

**AC-5: Meaningful Descriptions**
Given each blueprint,
When viewed in the blueprint browser,
Then name, description, and trade-off summary help users understand what the architecture is FOR and what it trades off.

## Architectural Acceptance Criteria

**AC-ARCH-LOC-1:** Blueprint files in `src/data/blueprints/*.yaml` (CREATE 8 files)
**AC-ARCH-PATTERN-1:** Follow existing format from whatsapp-messaging.yaml and telegram-messaging.yaml
**AC-ARCH-PATTERN-2:** Only reference components that exist in src/data/components/
**AC-ARCH-PATTERN-3:** Node positions laid out in readable left-to-right flow (x: 0-1200, y: 0-600)
**AC-ARCH-NO-1:** No new components — use existing 10 only
**AC-ARCH-NO-2:** No code changes — data files only

## Dev Notes

**Blueprint YAML format (following existing convention):**
```yaml
name: "E-Commerce Storefront"
description: "Request-response architecture with multi-tier caching for product catalog and checkout"
tier: 2
skeleton:
  schemaVersion: "2.0.0"
  nodes:
    - id: cdn-1
      componentId: cloudflare-cdn
      variantId: default
      position: { x: 0, y: 200 }
    # ... more nodes
  edges:
    - id: e1
      source: cdn-1
      target: nginx-1
    # ... more edges
```

**Trade-off diversity:** Each blueprint should highlight different trade-off profiles:
- E-commerce: performance + cost vs operational complexity
- Event-driven: throughput + scalability vs latency + complexity
- Real-time: real-time + DX vs cost + reliability at scale
- etc.

**V1 alignment:** Descriptions should help users understand WHY this architecture is structured this way and what it trades off — comprehension over blueprint shopping.
