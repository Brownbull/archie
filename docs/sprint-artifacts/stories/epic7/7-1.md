# Story 7-1: Data Context Types, Schema & Fit Engine

## Status: done
## Epic: Epic 7 -- Data Context Items

## Intent
**Epic Handle:** "Your data, scored against every component"
**Story Handle:** "This story teaches Archie what your data looks like and how to judge if a component is a good fit"

## Overview

As a user,
I want to define data context items (name, access pattern, size, structure type) on any component,
And have Archie evaluate each item's fit against the component's active variant,
So that I can see how well my actual data shapes match each component's strengths.

**FRs:** FR58, FR59, FR60, FR61, NFR13, NFR16
**Size:** MEDIUM (5 pts) -- 5 tasks, ~26 subtasks, 7 files
**Risk Flags:** DATA_PIPELINE, INPUT_SANITIZATION, ERROR_RESILIENCE, PURE_COMPONENT
**Depends on:** None (uses existing component library + architectureStore patterns)

## Acceptance Criteria

**AC-1: Data Context Item Type (FR58)**
Given the data context model,
When a data context item is defined,
Then it specifies: id (uuid), name (string), accessPattern (read-heavy | write-heavy | mixed | append-only), averageSize (small | medium | large | huge), structureType (simple-kv | nested-json | relational | binary-blob).

**AC-2: Fit Indicator Calculation (FR59)**
Given a data context item and a component's active variant,
When fit is evaluated,
Then a fit indicator is produced: great-fit | good-fit | trade-off | poor-fit | risky.

**AC-3: Fit Explanation (FR60)**
Given a fit indicator,
When expanded,
Then it shows the reason for the rating and contributing factors (access pattern compatibility, size handling, structure support).

**AC-4: Variant-Sensitive Fit (FR61)**
Given data context items on a component,
When the user switches configuration variant,
Then fit indicators re-evaluate against the new variant's characteristics.

**AC-5: Fit Calculation Performance (NFR13)**
Given up to 10 data context items per component,
When fit is calculated,
Then all indicators complete within 200ms.

**AC-6: Name Sanitization (NFR16)**
Given user-provided data context item names,
When stored and displayed,
Then names are sanitized (trimmed, HTML stripped, max 100 chars).

## Dev Notes

**Fit is derived, not stored.** DataContextItem definitions are persisted in YAML. FitResult is computed at runtime from current component library data. This means fit indicators auto-update if library data improves.

**Compatibility matrix:** The component library stores compatibility data per component type per variant. Example: Redis "cache" variant has { "read-heavy": "great", "write-heavy": "good", "simple-kv": "great", "relational": "poor" }. This data is AI-generated alongside other component metrics.

**Aggregation to FitLevel:**
- All positive -> great-fit
- Mostly positive -> good-fit
- Mixed -> trade-off
- Mostly negative -> poor-fit
- All negative -> risky
