# Story 7-2: Data Context UI -- Inspector Integration & Fit Display

## Status: done
## Epic: Epic 7 -- Data Context Items

## Intent
**Epic Handle:** "Your data, scored against every component"
**Story Handle:** "This story adds the data personality card to each component, showing how your data fits"

## Overview

As a user,
I want to add/edit/remove data context items from the component inspector and see fit indicators with expandable explanations,
So that I can evaluate how well my actual data shapes work with each component.

**FRs:** FR58, FR59, FR60, FR61, FR62
**Size:** MEDIUM (5 pts) -- 5 tasks, ~22 subtasks, 6 files
**Risk Flags:** PURE_COMPONENT, CROSS_STORE, ERROR_RESILIENCE, E2E_TESTING
**Depends on:** Story 7-1 (types + fit engine)

## Acceptance Criteria

**AC-1: Data Context Section in Inspector (FR58, FR62)**
Given a component selected on the canvas,
When the inspector opens,
Then a "Data Context" section shows existing data items with add/edit/remove controls.

**AC-2: Fit Indicator Display (FR59)**
Given data context items on a component,
When viewing the data context section,
Then each item shows a fit indicator badge (great-fit to risky) with a color-coded icon.

**AC-3: Expandable Explanation (FR60)**
Given a fit indicator,
When I click to expand,
Then I see the explanation text and contributing factors (positive/neutral/negative).

**AC-4: Variant Switch Updates Fit (FR61)**
Given data context items on a component,
When I switch the component's configuration variant,
Then all fit indicators re-evaluate and update.

**AC-5: CRUD Without Side Effects (FR62)**
Given data context items managed on a component,
When items are added, edited, or removed,
Then objective metrics are NOT affected -- data context is a separate analysis layer.

## Architectural Acceptance Criteria

**AC-ARCH-LOC-1:** DataContextPanel at `src/components/inspector/DataContextPanel.tsx`
**AC-ARCH-LOC-2:** FitIndicator at `src/components/inspector/FitIndicator.tsx`
**AC-ARCH-PATTERN-1:** DataContextPanel reads per-node data context from architectureStore
**AC-ARCH-PATTERN-2:** Fit indicators computed via useMemo calling evaluateFit -- derived, not stored
**AC-ARCH-PATTERN-3:** CRUD dispatches store actions -- does NOT import engine directly
**AC-ARCH-PATTERN-4:** FitIndicator is pure presentational -- receives FitResult as prop
**AC-ARCH-PATTERN-5:** Explanation uses same progressive disclosure pattern as metric explanations (FR32)
**AC-ARCH-NO-1:** No engine imports in UI components (evaluateFit called via useMemo wrapper, not directly)
**AC-ARCH-NO-2:** No modifications to recalculation pipeline -- data context is independent of metric scoring

## Dev Notes

**Fit computation in UI:** Use `useMemo` with dependencies on [dataContextItems, activeVariant, componentLibraryData]. When any dependency changes, all fit indicators recompute. This is cheap (O(items * factors), ~30 operations for 10 items).

**Progressive disclosure:** Same accordion/collapsible pattern used for metric explanations in the inspector. Fit factors display as a simple list with +/~/- icons.

**Color coding:** great-fit=emerald, good-fit=green, trade-off=amber, poor-fit=orange, risky=red. Consistent with heatmap severity progression.

**File path correction (2026-03-17):** Story originally referenced `ComponentInspector.tsx` which does not exist. Actual integration points: `ComponentDetail.tsx` (section) + `InspectorPanel.tsx` (nav entry). File Specification updated.

**Schema fix (2026-03-17):** Fixed pre-existing readonly tuple cast error in `architectureFileSchema.ts` (from Story 7-1) by replacing `as [T, ...T[]]` with `[...VALUES]` spread.

**Self-review fixes (2026-03-17):** Per-item testids on fit-badge and factor-icons, keyboard/ARIA support on FitIndicator, dynamic limit string.
