# Story 7-3: YAML Data Context Export/Import Round-Trip

## Status: drafted
## Epic: Epic 7 -- Data Context Items

## Intent
**Epic Handle:** "Your data, scored against every component"
**Story Handle:** "This story saves and restores the data personality cards through YAML"

## Overview

As a user,
I want my data context items saved when I export YAML and restored when I import,
So that my data definitions are portable and shareable alongside my architecture.

**FRs:** FR63, FR72, FR71
**Size:** SMALL (3 pts) -- 3 tasks, ~14 subtasks, 5 files
**Risk Flags:** DATA_PIPELINE, ERROR_RESILIENCE, INPUT_SANITIZATION
**Depends on:** Story 7-1 (schema + types)

## Acceptance Criteria

**AC-1: Export with Data Context Items (FR63)**
Given components have data context items defined,
When I export the architecture as YAML,
Then the exported file includes data_context per node with all item definitions.

**AC-2: Export Omits Empty Data Context (FR72)**
Given no data context items on any component,
When I export,
Then the data_context section is OMITTED entirely.

**AC-3: Import v2 YAML with Data Context (FR63)**
Given I import a v2 YAML with data_context items,
When the architecture loads,
Then data items are restored per-node and fit indicators are derived from current library data.

**AC-4: Import without Data Context**
Given I import a v2 or v1 YAML without data_context,
When the architecture loads,
Then no data context items exist, no fit indicators shown.

**AC-5: Import Invalid Data Context -- Rejection (FR71)**
Given a v2 YAML with invalid data context (unknown access pattern, missing name, extra fields),
When I import,
Then import fails with clear validation error message.

**AC-6: Fit Re-derivation on Import**
Given data context items imported,
When fit indicators are displayed,
Then fit is computed from current library data (not from stored fit results in YAML).

## Architectural Acceptance Criteria

**AC-ARCH-PATTERN-1:** Conditional export: include data_context only when nodes have items
**AC-ARCH-PATTERN-2:** Data context exported per-node in the node's YAML section (not a top-level section)
**AC-ARCH-PATTERN-3:** Fit results are NOT exported -- re-derived on import from current library data
**AC-ARCH-PATTERN-4:** Import pipeline extracts data context per node, sets in architectureStore
**AC-ARCH-PATTERN-5:** Snake_case in YAML (access_pattern, average_size, structure_type), camelCase in TS
**AC-ARCH-NO-1:** Data context NOT stored in Firestore -- per-architecture, per-YAML-file only
**AC-ARCH-NO-2:** No separate import function -- extends existing pipeline

## Dev Notes

**Per-node vs top-level:** Unlike weight_profile (top-level) and constraints (top-level array), data_context is PER-NODE. Each node in the YAML skeleton can have an optional `data_context` array. This matches the domain model -- data context items are per-component.

**Fit re-derivation:** Fit results depend on component library data which may change between export and import. Exporting fit results would create stale data. Instead, only the user's data definitions (the inputs) are persisted.

**Snake_case transform:** `accessPattern` -> `access_pattern`, `averageSize` -> `average_size`, `structureType` -> `structure_type`. Same Zod transform pattern.
