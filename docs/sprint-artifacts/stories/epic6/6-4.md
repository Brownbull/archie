# Story 6-4: YAML Constraint Export/Import Round-Trip

## Status: done
## Epic: Epic 6 -- Constraint Guardrails

## Intent
**Epic Handle:** "Your red lines, visible on the canvas"
**Story Handle:** "This story saves and restores the red lines by ensuring constraints round-trip through YAML"

## Overview

As a user,
I want my constraint definitions saved when I export YAML and restored when I import,
So that my threshold requirements are portable and shareable alongside my architecture.

**FRs:** FR57, FR72, FR71
**Size:** SMALL (3 pts) -- 3 tasks, ~14 subtasks, 5 files
**Risk Flags:** DATA_PIPELINE, ERROR_RESILIENCE
**Depends on:** Story 6-1 (schema), Story 6-2 (store)

## Acceptance Criteria

**AC-1: Export with Active Constraints (FR57)**
Given I have defined one or more constraints,
When I export the architecture as YAML,
Then the exported file includes a constraints section with all constraint definitions.

**AC-2: Export Omits Empty Constraints (FR72)**
Given no constraints are defined,
When I export,
Then the constraints section is OMITTED entirely.

**AC-3: Import v2 YAML with Constraints (FR57)**
Given I import a v2 YAML with constraints,
When the architecture loads,
Then constraints are restored in the store and violations are evaluated against loaded scores.

**AC-4: Import v2 YAML without Constraints**
Given I import a v2 YAML without constraints section,
When the architecture loads,
Then constraints default to empty array, no violations.

**AC-5: Import Invalid Constraints -- Rejection (FR71)**
Given a v2 YAML with invalid constraints (unknown category, threshold=5, missing operator),
When I import,
Then import fails with clear validation error message.

## Architectural Acceptance Criteria

**AC-ARCH-PATTERN-1:** Conditional export: include constraints only when array is non-empty
**AC-ARCH-PATTERN-2:** Constraint export uses snake_case field names in YAML (category_id, not categoryId)
**AC-ARCH-PATTERN-3:** Import pipeline extracts constraints and passes to loadArchitecture alongside nodes/edges/metadata/weightProfile
**AC-ARCH-PATTERN-4:** loadArchitecture sets constraints, triggers evaluation against loaded weighted scores
**AC-ARCH-NO-1:** Constraint data NOT stored in Firestore -- per-architecture, per-YAML-file only
**AC-ARCH-NO-2:** No separate import function for constraints -- extends existing pipeline

## Dev Notes

**Snake_case transform for constraints:** YAML uses `category_id`, TypeScript uses `categoryId`. The transform happens in the Zod schema (import) and in the exporter function (export). Follow the same pattern as weight_profile.

**Export button:** ExportButton.tsx reads from architectureStore and calls exporter -- needs to pass constraints alongside existing data (same pattern as Story 5-4 for weightProfile).
