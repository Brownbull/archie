# Story 6-2: Constraint Evaluation Engine & Store Integration

## Status: review
## Epic: Epic 6 -- Constraint Guardrails

## Intent
**Epic Handle:** "Your red lines, visible on the canvas"
**Story Handle:** "This story builds the referee that checks every score against the red lines and blows the whistle on violations"

## Overview

As a user,
I want constraint violations automatically detected after every recalculation,
So that I immediately know when my architecture crosses a threshold I care about.

**FRs:** FR52, FR56, NFR14
**Size:** MEDIUM (5 pts) -- 5 tasks, ~24 subtasks, 5 files
**Risk Flags:** DATA_PIPELINE, CROSS_STORE, PURE_COMPONENT
**Depends on:** Story 6-1 (types/schema), Story 5-2 (weighted engine + store)

## Acceptance Criteria

**AC-1: Constraint Evaluation (FR52)**
Given active constraints and weighted category scores,
When evaluateConstraints runs,
Then each constraint is checked: score vs threshold using the operator,
And violations list includes constraint ID, category, actual score, and threshold.

**AC-2: Synchronous Evaluation (NFR14)**
Given a recalculation triggers (component change or weight change),
When the pipeline runs,
Then constraint evaluation completes synchronously -- no separate loading state.

**AC-3: Per-Node Constraint Check (FR52)**
Given constraints apply to category scores,
When evaluating per-node,
Then each node's per-category scores are checked against all active constraints,
And violations are attributed to specific nodes.

**AC-4: CRUD Without Side Effects (FR56)**
Given constraints are added, edited, or removed,
When the constraint list changes,
Then constraint re-evaluation triggers,
But architecture nodes, edges, and metrics are NOT modified.

**AC-5: Store Integration**
Given the architectureStore,
When constraints are managed,
Then the store holds: constraints array, constraintViolations array, and CRUD actions.

## Architectural Acceptance Criteria

**AC-ARCH-LOC-1:** `evaluateConstraints` pure function in `src/engine/constraintEvaluator.ts`

**AC-ARCH-PATTERN-3:** Store action: addConstraint / updateConstraint / removeConstraint -- each triggers re-evaluation

**AC-ARCH-NO-1:** No React or Zustand imports in constraintEvaluator.ts
**AC-ARCH-NO-2:** No modifications to recalculationService.ts -- constraint evaluation happens in store after scoring
**AC-ARCH-NO-3:** Constraint CRUD does NOT trigger BFS propagation -- only re-evaluates existing scores

## Dev Notes

**Score-to-threshold mapping:** Category scores from `computeCategoryScores` are on a 1-10 scale (average of `numericValue` across metrics in the category). Constraints use the same scale. So "Cost lte 4" means "Cost category score must not exceed 4.0." Comparison is straightforward numeric.

**Pipeline position:** Constraint evaluation slots in AFTER weighted scoring. The pipeline becomes:
```
Component change -> BFS recalculation -> computedMetrics
  -> computeCategoryScores (unweighted)
  -> computeWeightedCategoryScores (weighted)
  -> evaluateConstraints (violations)
  -> heatmap + tier

Weight change (no BFS):
  -> read existing computedMetrics
  -> recompute weighted scores
  -> evaluateConstraints
  -> heatmap + tier

Constraint CRUD (no BFS, no recompute):
  -> read existing weighted scores
  -> evaluateConstraints
  -> update violations
```

**Performance (NFR14):** Constraint evaluation is O(constraints * nodes). For 10 constraints and 20 nodes = 200 comparisons. Well within synchronous budget.
