# Story 6-1: Constraint Schema & Types Foundation

## Status: done
## Epic: Epic 6 -- Constraint Guardrails

## Intent
**Epic Handle:** "Your red lines, visible on the canvas"
**Story Handle:** "This story defines the vocabulary for red lines by creating the types, schema, and validation rules for constraint definitions"

## Overview

As a user,
I want Archie to support constraint definitions in the YAML schema,
So that my hard threshold requirements can be saved, validated, and restored alongside my architecture.

**FRs:** FR51 (types), FR57 (schema), FR71 (validation)
**Size:** MEDIUM (4 pts) -- 4 tasks, ~20 subtasks, 6 files
**Risk Flags:** DATA_PIPELINE, INPUT_SANITIZATION, ERROR_RESILIENCE
**Depends on:** Story 5-1 (schema v2 foundation)

## Acceptance Criteria

**AC-1: Constraint Type Definition (FR51)**
Given the constraint type system,
When a constraint is created,
Then it specifies: id (uuid), categoryId (MetricCategoryId), operator ("lte" | "gte"), threshold (number 1-10, matching category score scale), label (user-provided string).

**AC-2: Schema Extension for Constraints (FR57)**
Given the v2 YAML schema,
When a constraints section is present,
Then each constraint validates against ConstraintSchema (category, operator, threshold, label).

**AC-3: Constraints Optional in Schema (FR72)**
Given a v2 YAML without a constraints section,
When the file is imported,
Then the constraints default to an empty array,
And the import succeeds.

**AC-4: Invalid Constraint Rejection (FR71, NFR17, NFR18)**
Given a v2 YAML with invalid constraints (unknown category, out-of-range threshold, missing fields),
When the file is imported,
Then it is rejected with clear Zod validation error messages.

**AC-5: Constraint Label Sanitization (NFR16)**
Given a constraint with a user-provided label,
When the label is stored,
Then it is trimmed to 100 characters max and stripped of HTML tags.

## Architectural Acceptance Criteria

**AC-ARCH-LOC-1:** Constraint types and constants in `src/lib/constants.ts`
**AC-ARCH-LOC-2:** ConstraintSchema in `src/schemas/architectureFileSchema.ts` -- both camelCase and snake_case variants
**AC-ARCH-PATTERN-1:** Constraint type uses `MetricCategoryId` for category reference -- same category enum as weights
**AC-ARCH-PATTERN-2:** Threshold is a number 1-10 matching the category score scale (from computeCategoryScores)
**AC-ARCH-PATTERN-3:** Operator is "lte" (must not exceed) or "gte" (must be at least) -- comparison against category score
**AC-ARCH-PATTERN-4:** Zod schema uses z.array(ConstraintSchema).optional() -- consistent with weight_profile pattern
**AC-ARCH-PATTERN-5:** YAML variant uses .transform() for snake_case to camelCase -- consistent with existing pattern
**AC-ARCH-NO-1:** Do NOT add constraint state to Zustand store -- that is Story 6-2
**AC-ARCH-NO-2:** Do NOT use z.record() for constraints -- use z.array(z.object()) for ordered, typed entries
**AC-ARCH-NO-3:** Do NOT modify engine files -- this story is schema/types only

## Dev Notes

**Threshold semantics:** Category scores from `computeCategoryScores` are on a 1-10 scale (matching individual metric `numericValue` range). A constraint "Cost lte 4" means "cost category score must not exceed 4.0." Threshold is a continuous number (not an integer enum) to allow fine-grained constraints like 6.5.

**Label sanitization:** Uses `sanitizeDisplayString()` from `src/lib/sanitize.ts` with `maxLength=100` — full 6-layer defense-in-depth pipeline (Unicode NFC, ReDoS guard, dangerous tag stripping, protocol URI removal, trim, truncate).

**Schema extension pattern:** Follows the same approach as weight_profile in Story 5-1 -- optional field, .strict() preserved, snake_case transform in YAML variant.

**Implementation note (2026-03-07):** ConstraintSchema (Zod) intentionally omits `id` field — id is on the Constraint interface for runtime use. Story 6-2 assigns ids when creating constraints. HydratedArchitecture uses `ParsedConstraint[]` (named alias for `Omit<Constraint, "id">`) to stay in sync.

**Review note (2026-03-07):** Deferred: error message quality assertions (assert Zod error message content is "clear" per AC-4, not just success boolean). LOW priority — current tests verify rejection, message content is Zod-generated and inherently descriptive.
