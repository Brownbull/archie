# Story 11-4: Demand + Failure Response Data for New Components

## Status: review
## Epic: Epic 11 -- Expanded Content Library

## Intent
**Epic Handle:** "More building blocks for more architectures"
**Story Handle:** "New components without demand/failure data are inert under simulation -- this story makes them react"

## Overview

As a user,
I want the 8 new components to respond to demand scenarios and failure scenarios realistically,
So that simulations reflect how AI, fintech, and data engineering components behave under stress and failure.

**FRs:** FR98
**Size:** MEDIUM (4 pts) -- 3 tasks, 14 subtasks, 8 files (data-only YAML modifications)
**Risk Flags:** DATA_PIPELINE, ERROR_RESILIENCE
**Depends on:** Story 11-1 and 11-2 (component YAML files exist), Story 9-1 (demand schema defined), Story 9-7 (failure schema defined)

## Acceptance Criteria

**AC-1: Demand Response Data for All 8 Components (FR98)**
Given the 8 new components from Stories 11-1 and 11-2,
When each is extended with a `demandResponses` section,
Then every component defines multiplicative modifiers for at least 2 demand variables where degradation is expected. Format matches existing component demand data.

**AC-2: Failure Response Data for All 8 Components (FR98)**
Given the 8 new components,
When each is extended with a `failureResponses` section,
Then every component defines metric-to-multiplier mappings for at least 3 of the 6 failure scenarios where degradation is expected.

**AC-3: Sparse Coverage Applied (FR98)**
Given components unaffected by certain demand/failure scenarios,
When no modifier is defined for that combination,
Then the engine defaults to 1.0 (no change), consistent with existing behavior.

**AC-4: Directional Accuracy**
Given all demand/failure responses,
When reviewed by domain experts,
Then modifiers reflect realistic directional trade-offs (e.g., LLM Gateway degrades heavily under traffic-spike on cost-efficiency and latency; Data Lake is resilient to traffic but degrades under data-corruption).

## Architectural Acceptance Criteria

**AC-ARCH-LOC-1:** All 8 new component YAML files in `src/data/components/` (MODIFY)

**AC-ARCH-PATTERN-3:** AI-generated data, hand-validated for directional accuracy

**Domain-specific degradation patterns:**
- LLM Gateway: extremely cost-sensitive under traffic (AI inference costs scale linearly), vulnerable to network partition (model API calls fail)
- Vector DB: query latency degrades with index size, resilient to traffic if index fits in memory
- Serverless: cold start latency under burst, cost scales linearly with traffic, resilient to node failure (managed)
- ETL Pipeline: throughput degrades with data size, vulnerable to data corruption (pipeline data loss)
- Payment Gateway: availability critical under burst (transaction timeouts), security-sensitive to data corruption
- Graph DB: traversal performance degrades with data size, vulnerable to node failure (consistency)
- Data Lake: resilient to traffic (append-only), vulnerable to data corruption (no ACID)
- SIEM: ingestion throughput degrades under traffic spike, alert latency increases with log volume
