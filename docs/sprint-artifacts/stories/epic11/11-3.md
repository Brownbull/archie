# Story 11-3: Cost-Efficiency Metric

## Status: done
## Epic: Epic 11 -- Expanded Content Library

## Intent
**Epic Handle:** "More building blocks for more architectures"
**Story Handle:** "Cost is the metric everyone asks about last but cares about first -- now it participates in scoring like every other dimension"

## Overview

As a user,
I want every component to have a cost-efficiency metric (1-10 scale),
So that I can see cost trade-offs in the heatmap, dashboard, and scoring pipeline alongside other architecture qualities.

**FRs:** FR97
**Size:** MEDIUM (3 pts) -- 3 tasks, 12 subtasks, 18 files (all YAML data modifications, no code)
**Risk Flags:** DATA_PIPELINE
**Depends on:** Story 11-1 and Story 11-2 (new component YAML files exist to modify)

## Acceptance Criteria

**AC-1: All Components Have Cost-Efficiency Metrics (FR97)**
Given all 18 components (10 existing + 8 new from 11-1, 11-2),
When their base_metrics are reviewed,
Then each includes at least one metric with `category: cost-efficiency`. Score semantics: 1-3 = expensive infrastructure, 4-6 = moderate cost, 7-10 = cost-efficient.

**AC-2: Per-Variant Cost Differentiation (FR97)**
Given each component's config variants,
When cost-efficiency metrics are reviewed,
Then variants with higher infrastructure requirements score lower on cost-efficiency than simpler variants (e.g., Redis Cluster costs more than Redis Standalone).

**AC-3: Weighted Scoring Participation (FR97)**
Given cost-efficiency metrics exist on components,
When the weight profile assigns cost-efficiency weight > 0,
Then cost-efficiency participates in weighted scoring, heatmap computation, and dashboard category scores identically to other metric categories.

**AC-4: Heatmap Sensitivity (FR97)**
Given the weight profile with cost-efficiency set to maximum weight (1.0) and other categories reduced,
When the heatmap renders,
Then components with poor cost-efficiency (score 1-3) show as warning/bottleneck status.

**AC-5: No Schema Changes Required**
Given the existing `METRIC_CATEGORIES` constant at `src/lib/constants.ts` line 74,
When `cost-efficiency` is checked,
Then it already exists as a category (`id: "cost-efficiency"`, `shortName: "Cost"`). No schema or constant changes are needed -- only data (YAML) additions.

## Dev Notes

**Key discovery:** The `cost-efficiency` category already exists in `METRIC_CATEGORIES` (line 74 of constants.ts: `{ id: "cost-efficiency", name: "Cost Efficiency", shortName: "Cost" }`). The `DEFAULT_WEIGHT_PROFILE` auto-includes it since it's built from `METRIC_CATEGORIES.map()`. The entire scoring pipeline already handles it. The ONLY thing missing is actual data in the component YAML files.

**Metric format in YAML:**
```yaml
- id: cost-efficiency
  value: high      # or medium or low
  numeric_value: 8 # 1-10 scale
  category: cost-efficiency
```

**Note on Stories 11-1/11-2:** Stories 11-1 and 11-2 were completed BEFORE this story, and already included cost-efficiency metrics. Task 2 was pre-done. Only 10 existing components needed modification.

**Variant ID discrepancies (resolved):** Story subtasks used informal names. Actual variant IDs used: `citus-distributed` (not `distributed`), `static-caching` (not `static-assets`), `simple-cache` (not `local-cache`), `single-server` (not `single-instance`), `standalone` (not `single-node` for Prometheus). Kafka/node-express/rabbitmq each had 2 variants (not 3 as listed) — extra variant values skipped per AC-ARCH-NO-2.
