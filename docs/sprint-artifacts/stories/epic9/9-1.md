# Story 9-1: Demand Schema & Scenario Preset Definitions

## Status: done
## Epic: Epic 9 -- Demand Simulation

## Intent
**Epic Handle:** "See your architecture under pressure"
**Story Handle:** "This story teaches Archie what demand looks like by defining the variables, levels, and named scenarios"

## Overview

As a user,
I want the system to have a formal model of demand conditions (traffic volume, data size, concurrent users, geographic spread, burst pattern),
So that components can express how their metrics shift under different operating conditions.

**FRs:** FR78, FR79
**Size:** MEDIUM (4 pts) -- 4 tasks, ~16 subtasks, 5 files
**Risk Flags:** DATA_PIPELINE, INPUT_SANITIZATION
**Depends on:** None (foundational types, no runtime code)

## Acceptance Criteria

**AC-1: Demand Variable Definitions (FR78)**
Given the demand simulation data model,
When demand variables are defined,
Then 5 demand dimensions are available: traffic-volume, data-size, concurrent-users, geographic-spread, burst-pattern, each with 3-4 ordered levels (e.g., low/medium/high/extreme).

**AC-2: Demand Response Schema (FR78)**
Given the component data model,
When a demandResponses section is defined on a component,
Then it specifies multiplicative modifiers (0.0-1.0 as fraction of base) per metric per demand variable per level, validated by Zod schema.

**AC-3: Scenario Preset Definitions (FR79)**
Given the 6 MVP scenario presets (Traffic Peak, Cost Optimized, Security First, Startup MVP, Enterprise Production, High Availability),
When a scenario is defined,
Then it specifies: id, name, description, icon, and a demand profile mapping each variable to a level.

**AC-4: Scenario Preset Data (FR79)**
Given the 6 scenario presets,
When all are loaded,
Then each provides a complete demand profile (all 5 variables set to specific levels) with a user-facing description explaining the operating context.

**AC-5: Validation (NFR21)**
Given demand response data or scenario presets,
When loaded or imported,
Then invalid entries (multiplier outside 0.1-1.0, unknown variable/level, incomplete profile) are rejected with descriptive errors.

## Dev Notes

**Multiplier semantics:** 1.0 = no change from base metric. 0.7 = 70% of base (30% degradation). Values below 1.0 always represent degradation. No multiplier above 1.0 in MVP (demand doesn't improve components). Floor at 0.1 prevents zeroing out metrics.

**Per-component scope (MVP):** demandResponses live at the component level, not per-variant. Per-variant demand responses are deferred to Enterprise tier — see ADR in project memory.

**Scenario YAML format:**
```yaml
id: traffic-peak
name: Traffic Peak
description: "Extreme traffic surge — 10x normal volume with periodic spikes"
icon: TrendingUp
demandProfile:
  traffic-volume: extreme
  concurrent-users: high
  burst-pattern: periodic-spikes
  data-size: medium
  geographic-spread: single-region
```
