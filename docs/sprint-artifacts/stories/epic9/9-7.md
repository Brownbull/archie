# Story 9-7: Failure Scenario Presets

## Status: done
## Epic: Epic 9 -- Demand Simulation

## Intent
**Epic Handle:** "See your architecture under pressure"
**Story Handle:** "This story asks 'what breaks?' using the same engine that asks 'what stresses?' -- failure is just demand with sharper teeth"

## Overview

As a user,
I want to select failure scenarios (node failure, network partition, database failure, etc.) alongside demand scenarios,
So that I can see how my architecture degrades under both operational pressure and failure conditions simultaneously.

**FRs:** FR91, FR92
**Size:** MEDIUM (5 pts) -- 5 tasks, 22 subtasks, 8 files
**Risk Flags:** DATA_PIPELINE, CROSS_STORE, INPUT_SANITIZATION
**Depends on:** Story 9-3 (demand engine exists), Story 9-4 (scenario selector UI pattern established)

## Acceptance Criteria

**AC-1: 6 Failure Preset Definitions (FR91)**
Given the failure scenario data model,
When all presets are loaded,
Then 6 failure presets are available: Single Node Failure, Network Partition, Database Failure, Traffic Spike (10x), Region Outage, Data Corruption. Each defines metric-to-multiplier mappings using the same format as demand responses.

**AC-2: Failure Dropdown (FR92)**
Given the canvas toolbar,
When the failure scenario selector is visible,
Then it shows "No Failure" (default) plus 6 failure presets alongside (but separate from) the demand scenario dropdown. Both can be active simultaneously.

**AC-3: Same Engine, Same Format (FR91)**
Given the demand engine from Story 9-3,
When a failure scenario is applied,
Then it uses the same `applyDemandModifiers` engine with failure-specific multipliers applied at Level 4 (after demand modifiers at Level 3). The data format is identical to demand scenarios.

**AC-4: Stacking with Demand Scenarios**
Given both a demand scenario (e.g., Traffic Peak) and a failure scenario (e.g., Database Failure) active simultaneously,
When metrics are recalculated,
Then demand modifiers apply first (Level 3), then failure modifiers stack multiplicatively on top (Level 4). Final values clamped to [1.0, 10.0].

**AC-5: Performance Budget (NFR24)**
Given 15 components with both demand and failure scenarios active,
When recalculation runs,
Then combined demand + failure computation completes within 20ms.

**AC-6: YAML Persistence**
Given an active failure scenario,
When the architecture is exported,
Then the active failure scenario ID is included in the YAML header alongside the demand scenario ID. On import, both are restored if matching known presets.

**AC-7: Enum-Constrained Failure Preset IDs (Security -- INPUT_SANITIZATION)**
Given failure preset selection,
When a failure ID is received from the UI or YAML import,
Then it is validated against a strict enum of known preset IDs. Unknown IDs are rejected and default to null (no failure).

## Dev Notes

**Failure is demand with sharper teeth.** Same engine, same data format, different multiplier targets. Demand modifiers are per-variable/per-level; failure modifiers are per-metric (simpler). Both stack multiplicatively.

**Pipeline ordering (corrected from competitive analysis):**
```
Base metrics → Config variant → Demand modifiers (L3) → Failure modifiers (L4) → Weighted scoring → Output consumers
```

**Security note (from hardening analysis):** Failure preset IDs must be validated against a strict enum. Never accept arbitrary strings from YAML import or UI input as failure preset IDs. Unknown IDs → null (no failure active).
