# Story 9-3: Demand Simulation Engine

## Status: done
## Epic: Epic 9 -- Demand Simulation

## Intent
**Epic Handle:** "See your architecture under pressure"
**Story Handle:** "This story builds the pure engine that applies demand pressure to component metrics — same inputs, same outputs, every time"

## Overview

As a user,
I want the system to compute effective metrics by applying demand modifiers to base component metrics,
So that selecting a scenario produces deterministic, realistic degradation across all components on my canvas.

**FRs:** FR81, FR82
**Size:** MEDIUM (5 pts) -- 4 tasks, ~18 subtasks, 4 files
**Risk Flags:** PURE_COMPONENT, DATA_PIPELINE
**Depends on:** Story 9-1 (schemas), Story 9-2 (demand data)

## Acceptance Criteria

**AC-1: Demand Modifier Application (FR81)**
Given a component with base metrics and demandResponses,
When `applyDemandModifiers()` is called with a scenario's demand profile,
Then each metric is multiplied by the corresponding modifier for each active demand variable, stacking multiplicatively (e.g., base 7 × 0.7 × 0.9 = 4.41).

**AC-2: Missing Modifier Defaults (FR81)**
Given a component that does not define a demandResponse for a specific variable/level combination,
When demand modifiers are applied,
Then the missing modifier defaults to 1.0 (no change).

**AC-3: Multi-Variable Stacking (FR81)**
Given a scenario with 3+ demand variables at non-default levels,
When modifiers are stacked multiplicatively,
Then the result is clamped to floor 1.0 and ceiling 10.0 on the final numericValue (after all modifiers applied).

**AC-4: Batch Application (FR82)**
Given all components on the canvas,
When a scenario is selected,
Then `computeDemandAdjustedMetrics()` returns a Map<nodeId, AdjustedMetric[]> with effective metrics for every component.

**AC-5: Performance (NFR22)**
Given 15 components on the canvas with demand responses,
When a scenario is applied,
Then computation completes within 20ms.

**AC-6: No-Scenario Pass-Through**
Given no active scenario (null/undefined),
When demand modifiers are computed,
Then base metrics are returned unchanged (identity operation).

## Dev Notes

**Multiplicative stacking rationale:** Additive deltas (base - 3 - 2 = 2) can produce zero or negative values. Multiplicative (base × 0.7 × 0.8 = 3.92) naturally produces diminishing returns and never reaches zero. This is more realistic: the second stress factor degrades less because the system is already under load.

**Floor at 1.0 (not 0):** A metric at 1.0 is "terrible but functional." At 0 it's meaningless. Archie's directional scale is 1-10; 0 would break display logic and mental models.

**Integration point:** Story 9-4 will wire this engine into the recalculation pipeline. The demand engine sits BETWEEN config variant resolution and weighted scoring:
```
Base metrics → Config variant → Demand modifiers (NEW) → Weighted scoring → Heatmap
```
