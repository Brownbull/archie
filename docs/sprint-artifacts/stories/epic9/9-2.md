# Story 9-2: Component Demand Response Data

## Status: done
## Epic: Epic 9 -- Demand Simulation

## Intent
**Epic Handle:** "See your architecture under pressure"
**Story Handle:** "This story fills in how each component reacts to demand — the data that makes simulations meaningful"

## Overview

As a user,
I want each component in the library to express how its metrics shift under different demand conditions,
So that selecting a scenario shows realistic degradation patterns across my architecture.

**FRs:** FR80
**Size:** MEDIUM (4 pts) -- 3 tasks, ~14 subtasks, 10 files
**Risk Flags:** DATA_PIPELINE, ERROR_RESILIENCE
**Depends on:** Story 9-1 (demand schema and types)

## Acceptance Criteria

**AC-1: All 10 Components Have Demand Response Data (FR80)**
Given the 10 existing component YAML files,
When each is extended with a demandResponses section,
Then every component defines multiplicative modifiers for at least 2 demand variables where degradation is expected.

**AC-2: Sparse Coverage (FR80)**
Given a component that is unaffected by a demand variable (e.g., CDN is unaffected by data-size:small),
When no modifier is defined for that variable/level combination,
Then the engine treats the missing entry as 1.0 (no change). Components are NOT required to define responses for all 5×4=20 cells.

**AC-3: Cross-Component Consistency**
Given components of the same category (e.g., Redis Cache vs Redis Data Store),
When both define demand responses for traffic-volume:extreme,
Then the cache component degrades less on throughput than the data store (caching is purpose-built for traffic), reflecting real-world behavior.

**AC-4: Schema Validation**
Given the extended component YAML files,
When loaded through the component library pipeline,
Then all pass DemandResponseSchema validation without errors.

## Architectural Acceptance Criteria

**AC-ARCH-LOC-1:** Component YAML files in `src/data/components/*.yaml` (MODIFY all 10)
**AC-ARCH-LOC-2:** Component schema extended in `src/schemas/componentSchema.ts` (MODIFY — add optional demandResponses field)
**AC-ARCH-PATTERN-1:** demandResponses is OPTIONAL on ComponentSchema — components without it default to 1.0 everywhere
**AC-ARCH-PATTERN-2:** Multipliers are sparse — only non-1.0 values need to be defined
**AC-ARCH-PATTERN-3:** AI-generated data, hand-validated for directional accuracy
**AC-ARCH-NO-1:** No engine logic — data only
**AC-ARCH-NO-2:** No UI changes

## Dev Notes

**Data generation approach:** Use AI (Opus) to generate demand responses for each component, then hand-validate for directional accuracy. The data should reflect real-world behavior: Nginx handles traffic better than Node.js, Redis Cache handles burst better than PostgreSQL, etc.

**Sparse format example (postgresql.yaml):**
```yaml
demandResponses:
  traffic-volume:
    high:
      throughput: 0.7
      latency: 0.8
    extreme:
      throughput: 0.5
      latency: 0.6
      scalability: 0.6
      operational-complexity: 0.8
  data-size:
    large:
      query-performance: 0.7
      cost-efficiency: 0.8
    massive:
      query-performance: 0.5
      cost-efficiency: 0.6
      storage-efficiency: 0.7
  concurrent-users:
    high:
      connection-pooling: 0.6
      throughput: 0.8
```

**V2 (Honest Uncertainty) note:** This data is AI-generated and directional. It does NOT represent benchmarked performance. The UI should make this provenance clear when demand simulation is active.
