# Story 8-1: Stack Data Model & Library Integration

## Status: review
## Epic: Epic 8 -- Stack Browsing & Placement

## Intent
**Epic Handle:** "Pre-wired building blocks, one drag away"
**Story Handle:** "This story teaches the library what a stack looks like by defining the data model and loading stack definitions"

## Overview

As a user,
I want Archie to load pre-composed technology stack definitions from the component library,
So that I can browse complete architectural patterns instead of assembling components one by one.

**FRs:** FR68, FR65 (data layer)
**Size:** MEDIUM (4 pts) -- 4 tasks, ~18 subtasks, 5 files
**Risk Flags:** DATA_PIPELINE, ERROR_RESILIENCE
**Depends on:** None (uses existing component library infrastructure)

## Acceptance Criteria

**AC-1: Stack Definition Type (FR68)**
Given the component library data model,
When stack definitions are loaded,
Then each stack specifies: id, name, description, components (with positions), connections (pre-wired), and a trade-off summary.

**AC-2: Library Loading (FR68)**
Given stack definitions stored alongside component data in Firestore,
When the component library initializes,
Then stacks are loaded and cached with the same pattern as components.

**AC-3: Trade-off Profile (FR65)**
Given a stack definition,
When the trade-off profile is computed,
Then it aggregates the component metrics into a summary (per-category average across stack components).

**AC-4: Validation (NFR18)**
Given stack definitions from the library,
When loaded,
Then each stack validates against the StackDefinition schema (required fields, valid component refs, valid connections).

## Architectural Acceptance Criteria

**AC-ARCH-LOC-1:** Stack types in `src/lib/constants.ts`
**AC-ARCH-LOC-2:** Stack schema in `src/schemas/stackSchema.ts` (or extend architectureFileSchema)
**AC-ARCH-LOC-3:** Stack loading in `src/services/componentLibrary.ts`
**AC-ARCH-PATTERN-1:** StackDefinition: { id, name, description, components: StackComponent[], connections: StackConnection[], tradeOffProfile: CategoryScore[] }
**AC-ARCH-PATTERN-2:** StackComponent: { componentId (library ref), variantId, relativePosition: {x, y} }
**AC-ARCH-PATTERN-3:** Same caching pattern as component library -- load from Firestore, cache in memory
**AC-ARCH-PATTERN-4:** Trade-off profile is pre-computed in library data (not computed at browse time)
**AC-ARCH-NO-1:** No canvas/React Flow code in this story -- data model only
**AC-ARCH-NO-2:** No modifications to architectureStore -- stacks are library data, not architecture state

## Dev Notes

**Stack data is LIBRARY data, not user data.** Stacks are pre-defined architectural patterns loaded from Firestore (same as component definitions). Users browse and place them but don't create custom stacks in Phase 2.

**Trade-off profile:** Pre-computed per-category averages stored in the stack definition. This avoids computing on the fly during browsing. Updated when component library data changes.

**Connection indices:** StackConnection uses indices into the components array (0-based) rather than component IDs, since the same component type could appear multiple times in a stack.

**Implementation notes (2026-03-10):** Upgraded existing flat schema (`componentIds: string[]`) to rich model. Backward-compat aliases (`StackSchema`, `Stack`) preserved for `stackRepository.ts`. Self-review finding: removed duplicate `StackDefinition` interface from constants.ts, narrowed `categoryId` schema to `MetricCategoryId` enum. 38 schema tests + 27 library tests all green.
