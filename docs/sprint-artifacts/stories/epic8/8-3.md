# Story 8-3: Stack Canvas Placement & Post-Placement Editing

## Status: drafted
## Epic: Epic 8 -- Stack Browsing & Placement

## Intent
**Epic Handle:** "Pre-wired building blocks, one drag away"
**Story Handle:** "This story drops the pre-wired block onto the canvas and lets users rewire individual pieces"

## Overview

As a user,
I want to drag a stack from the Stacks tab onto the canvas and have all components placed with pre-wired connections in a single action,
And after placement I want to edit individual stack components independently.

**FRs:** FR66, FR67, NFR15
**Size:** MEDIUM (5 pts) -- 5 tasks, ~22 subtasks, 6 files
**Risk Flags:** CROSS_STORE, DATA_PIPELINE, ERROR_RESILIENCE, PURE_COMPONENT
**Depends on:** Story 8-1 (data model), Story 8-2 (drag data)

## Acceptance Criteria

**AC-1: Drag-and-Drop Placement (FR66)**
Given a stack card in the Stacks tab,
When I drag it onto the canvas,
Then all included components are placed at calculated positions with their pre-defined connections as a single action.

**AC-2: Placement Performance (NFR15)**
Given a stack with up to 6 components,
When dropped on the canvas,
Then all components render with connections within 500ms.

**AC-3: Individual Editability (FR67)**
Given stack components placed on the canvas,
When I interact with any placed component,
Then I can: change configuration variants, add/remove connections, reposition independently, swap components using the in-place selector.

**AC-4: Recalculation Trigger**
Given stack components placed,
When placement completes,
Then the full recalculation pipeline runs (BFS + weighted + constraints + heatmap + tier).

**AC-5: Position Calculation**
Given a drop position on the canvas,
When placing a multi-component stack,
Then components are arranged using relative positions from the stack definition, offset by the drop point.

## Architectural Acceptance Criteria

**AC-ARCH-PATTERN-1:** Canvas drop handler reads 'application/archie-stack' drag data type
**AC-ARCH-PATTERN-2:** Stack placement creates individual nodes and edges in architectureStore -- no special "stack group" concept
**AC-ARCH-PATTERN-3:** After placement, stack components are indistinguishable from manually placed components
**AC-ARCH-PATTERN-4:** Position calculation: dropPoint + relativePosition for each component
**AC-ARCH-PATTERN-5:** Single batch operation: addNodes + addEdges + triggerRecalculation (not individual adds)
**AC-ARCH-NO-1:** No "stack group" node type -- placed components are regular ArchitectureNodes
**AC-ARCH-NO-2:** No undo-stack-placement feature -- standard undo handles individual operations
**AC-ARCH-NO-3:** No stack placement preview/ghost -- direct placement on drop

## Dev Notes

**No stack grouping:** After placement, stack components are independent nodes. There's no "ungroup" operation because they were never grouped. This simplifies the model -- stacks are just a convenient placement tool.

**Batch operation for performance (NFR15):** Using a single state update (set nodes + set edges) instead of individual addNode calls prevents N re-renders. The single triggerRecalculation at the end handles the full pipeline.

**Component hydration:** resolveStackPlacement uses componentLibrary to hydrate each StackComponent reference into a full node with metrics, config options, etc. Same hydration as individual component placement.
