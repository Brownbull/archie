# Story 7.5-2: Pathway Suggestions Hook & Store Integration

## Status: done
## Epic: Epic 7.5 -- Weighted Pathway Guidance

## Intent
**Epic Handle:** "Show the roads, not just the dot"
**Story Handle:** "This story wires the pathway engine to the store so the UI can read suggestions reactively"

## Overview

As a user,
I want pathway suggestions to recompute automatically when I change weights, constraints, components, or data context,
So that the suggestions always reflect my current architecture state.

**FRs:** FR73, FR74, FR75, FR76, FR77
**Size:** SMALL (3 pts) -- 3 tasks, ~10 subtasks, 3 files
**Risk Flags:** CROSS_STORE
**Depends on:** Story 7.5-1 (pathwayEngine.ts must exist)

## Acceptance Criteria

**AC-1: Reactive Computation**
Given the architecture store state (tier, weights, constraints, data context items, nodes),
When any of these change,
Then the `usePathwaySuggestions` hook recomputes suggestions via `computePathwaySuggestions`.

**AC-2: No New Store State**
Given pathway suggestions are derived data,
When computed,
Then they live in a `useMemo` inside the hook -- not as new fields in `architectureStore`.

**AC-3: Input Assembly**
Given the hook reads from `architectureStore` and `componentLibrary`,
When assembling inputs for `computePathwaySuggestions`,
Then it passes: tier gaps, next tier requirements, all components, existing node categories, existing component IDs, weight profile, constraints, data context items, and category scores.

**AC-4: Empty State**
Given no tier evaluation (no components on canvas) or max tier achieved,
When the hook runs,
Then it returns an empty suggestion array (not null, not undefined).

## Architectural Acceptance Criteria

**AC-ARCH-LOC-1:** Hook in `src/hooks/usePathwaySuggestions.ts` (CREATE)
**AC-ARCH-PATTERN-1:** Hook uses `useMemo` over `computePathwaySuggestions` -- no debounce, no worker
**AC-ARCH-PATTERN-2:** Hook reads from `architectureStore` via selectors (useShallow where needed)
**AC-ARCH-PATTERN-3:** `componentLibrary.getAllComponents()` called INSIDE useMemo (revised). Rationale: `Array.from(this.components.values())` creates a new array reference on every call, which defeats memoization if called outside and passed as a dep. The service is a singleton cache loaded once at app init â€” intentionally non-reactive. Omitting from dep array is deliberate.
**AC-ARCH-NO-1:** No new state fields added to architectureStore
**AC-ARCH-NO-2:** No side effects in the hook -- pure derivation only
