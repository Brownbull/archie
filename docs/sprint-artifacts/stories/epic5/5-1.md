# Story 5-1: YAML Schema v2 -- Weight Profile Foundation

## Status: done
## Epic: Epic 5 -- Priority Scoring

## Intent
**Epic Handle:** "Your priorities reshape the scoreboard"
**Story Handle:** "This story builds the mixing board's memory by defining how slider positions are stored, validated, and migrated"

## Overview

As a user,
I want Archie to support a v2 YAML schema that includes weight profiles,
So that my priority settings can be saved and restored when importing/exporting architecture files.

**FRs:** FR69, FR70, FR71, NFR18, NFR19
**Size:** MEDIUM (4 pts) -- 4 tasks, ~22 subtasks, 7 files
**Risk Flags:** DATA_PIPELINE, ERROR_RESILIENCE, INPUT_SANITIZATION

## Acceptance Criteria

**AC-1: Schema Version Bump**
Given the schema version is bumped to "2.0.0",
When the app exports or imports YAML,
Then the schema_version field reads "2.0.0".

**AC-2: v1-to-v2 Migration (FR70)**
Given a v1 YAML file is imported,
When the importer detects a version mismatch (major version 1 vs app version 2),
Then it runs the "1" migration adding a default weight profile (all 7 categories = 1.0),
And the import succeeds with the migrated data passing v2 schema validation.

**AC-3: Weight Profile Validation (FR71, NFR18, NFR19)**
Given a v2 YAML file with a weight_profile field,
When the importer validates the file,
Then each weight must be a number between 0 and 1 (inclusive),
And the weight_profile must contain exactly the 7 known category IDs,
And invalid weight profiles are rejected with clear Zod error messages.

**AC-4: Missing Weight Profile Defaults**
Given a v2 YAML file WITHOUT a weight_profile field,
When the file is imported,
Then the weight profile defaults to equal weights (all 7 categories = 1.0),
And the import succeeds.

**AC-5: Strict Schema Enforcement**
Given ArchitectureFileSchema uses .strict(),
When v2 fields are added to the schema shape,
Then the schema uses .strict() on the v2 shape (no passthrough of unknown keys),
And the YAML variant applies the standard snake_case-to-camelCase transform for weight_profile.

## Dev Notes

**CORRECTED (Story 5-1 implementation):** Migration operates on POST-transform camelCase data (`ArchitectureFile`). The pipeline runs safeParse first (snake_case → camelCase), then version check, then migration. So migration adds `weightProfile` (camelCase), not `weight_profile`.

**Category IDs use kebab-case** (e.g., `operational-complexity`). Same in both YAML and TypeScript -- no transform needed for keys within weight_profile. Only the top-level field transforms: `weight_profile` (YAML) -> `weightProfile` (TS).

**Weight semantics:** MULTIPLICATIVE. Default=1.0 is identity (no change). Range 0-1. UI clamps minimum to 0.1 (Story 5-3), but schema accepts full 0-1 range. A weight of 0.0 means "completely ignore this category."

**checkSchemaVersion behavior:** With CURRENT_SCHEMA_VERSION bumped to "2.0.0", importing a v1 file returns `{ status: "migrate", migrationKey: 1 }`. The migration dispatch path already exists.
