## Overview

As a user,
I want the scoring dashboard, heatmap, and tier system to reflect my weight preferences,
So that I see my architecture evaluated through my own priorities.

## Acceptance Criteria

**AC-1: Weighted Category Scores (FR45, FR49)**
Given a weight profile is set in the architecture store,
When computeCategoryScores runs and results are passed to computeWeightedCategoryScores,
Then each category score is multiplied by its corresponding weight,
And both weighted and unweighted scores are available in the store.

**AC-2: Weighted Heatmap (FR46)**
Given the heatmap calculator receives weighted per-node scores,
When computing node heatmap status,
Then the heatmap reflects the weighted per-node overall score.

**AC-3: Weighted Tier Evaluation (FR47)**
Given the tier evaluator receives weighted category scores,
When evaluateTier runs,
Then min_category_score requirements check against weighted scores.

**AC-4: Recalculation Performance (NFR12)**
Given a weight profile slider changes,
When recalculation triggers,
Then dashboard, heatmap, and tier update within 200ms,
And weight changes do NOT trigger BFS propagation -- only scoring recomputation.

**AC-5: Store Integration**
Given evaluateAndSetTier currently passes unweighted scores,
When weights are available in the store,
Then it passes weighted category scores to evaluateTier.

## Architectural Acceptance Criteria

**AC-ARCH-LOC-1:** Weighted functions in existing engine files: `computeWeightedCategoryScores` in dashboardCalculator.ts, `computeWeightedNodeScore` in heatmapCalculator.ts
**AC-ARCH-LOC-2:** Weight profile state in `src/stores/architectureStore.ts` -- not a separate store
**AC-ARCH-PATTERN-1:** Weighted functions are pure -- weights passed as parameters, no store/service imports in engine/
**AC-ARCH-PATTERN-2:** Weight profile stored in architectureStore alongside nodes, edges, computedMetrics
**AC-ARCH-PATTERN-3:** `computeWeightedAggregateScore`: sum(score*weight)/sum(weight), falls back to equal weights if sum(weight)===0
**AC-ARCH-PATTERN-4:** `setWeightAndRecalculate` bypasses recalculationService.run() -- only recomputes scoring layer (O(nodes), not O(graph))
**AC-ARCH-PATTERN-5:** Both triggerRecalculation and setWeightAndRecalculate produce consistent weighted results
**AC-ARCH-NO-1:** No separate weightStore -- weight profile is architecture domain data
**AC-ARCH-NO-2:** No Zustand store imports inside engine/ directory
**AC-ARCH-NO-3:** No modifications to recalculationService.ts -- weighting happens downstream
**AC-ARCH-NO-4:** No BFS propagation triggered on weight changes

## Dev Notes

**Pipeline after Epic 5:**
```
Component change -> recalculationService.run() (BFS) -> computedMetrics
  -> computeCategoryScores (unweighted)
  -> computeWeightedCategoryScores (weighted)
  -> weighted scores -> heatmap + tier

Weight slider change (no BFS):
  -> read existing computedMetrics
  -> computeCategoryScores (unweighted)
  -> computeWeightedCategoryScores (weighted)
  -> weighted scores -> heatmap + tier
```

**Performance (NFR12):** Weight changes are O(nodes * categories). For 20 components with 7 categories = ~140 multiplications + 20 heatmap lookups + 1 tier evaluation. Well within 200ms.

**Future pipeline insertion point:** Epic 6 (constraints) inserts between weighted scores and heatmap/tier.
